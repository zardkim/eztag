## 1. 개요 및 현재 vs 새로운 흐름

### 1-1. 현재 흐름 (직접 편집 모드)

```
사이드바 (라이브러리 폴더 트리)
    → 폴더 클릭
    → 파일 목록 표시
    → 태그 편집 → 즉시 파일에 반영
    → LRC 검색 → 즉시 파일에 반영
    → 파일명 변경 → 즉시 파일에 반영
    → (완료 후) 이동 버튼 → 대상 폴더로 이동
```

**문제점:**
| 항목 | 문제 |
|------|------|
| 즉시 반영 | 실수 시 되돌리기 어려움 |
| 사이드바 = 라이브러리 트리 | 폴더 탐색과 작업 공간이 혼재 |
| 작업 범위 불명확 | "어디까지 작업했는지" 파악 어려움 |
| 히스토리 없음 | 과거 작업 내역 추적 불가 |

## 2. 핵심 개념 정의

### 2-1. 워크스페이스 (Workspace)

- 현재 편집 세션에서 불러온 파일들의 집합
- 파일을 물리적으로 복사하지 않음 (원본 경로 참조)
- 변경 사항(태그, 파일명 등)을 **임시 저장**하고, "적용" 시 일괄 반영
- 여러 폴더/파일을 한 세션에 묶어 작업 가능
- 세션 완료 시 히스토리로 저장

### 2-2. 스테이징 (Staging)

- Git의 Staging Area 개념과 유사
- 태그 편집 → DB에 `pending_tags` 저장 (파일은 아직 미변경)
- LRC 검색 → LRC 파일 즉시 저장 (파일 저장이 안전한 작업이므로 예외)
- 파일명 변경 → `pending_rename` 저장 (파일은 아직 미변경)
- "적용" 클릭 → 모든 pending 변경 사항을 실제 파일에 일괄 write

### 2-3. 히스토리 (History)

- 세션 단위로 기록: 언제, 어떤 파일, 무슨 작업
- 세션 상태: `editing` → `applied` → (선택) `moved`
- 히스토리에서 과거 세션 조회 및 파일 상태 확인 가능

- 현재와 동일한 테이블/카드 뷰 유지
- 각 파일에 **변경 상태 배지** 추가:
  - 🟡 `변경됨` - pending 변경 있음

  - ⚪ `원본` - 변경 없음
- 파일 선택 시 우측 태그 패널에서 편집 (현재와 동일)
- 저장 버튼 → 즉시 파일 반영 대신 **스테이징** (DB 임시 저장)

```sql
id          SERIAL PRIMARY KEY
status      VARCHAR(20) NOT NULL DEFAULT 'editing'
            -- 'editing' | 'applied' | 'moved' | 'discarded'
name        VARCHAR(200)                  -- 사용자 지정 이름 (선택)
created_at  TIMESTAMP NOT NULL DEFAULT NOW()
```

#### `workspace_items` (세션 내 파일)
```sql
id              SERIAL PRIMARY KEY
session_id      INT NOT NULL REFERENCES workspace_sessions(id) ON DELETE CASCADE
file_path       TEXT NOT NULL             -- 원본 파일 경로
original_tags   JSONB                     -- 불러올 당시 태그 스냅샷
pending_tags    JSONB                     -- 변경 예정 태그 (staged)
pending_rename  TEXT                      -- 변경 예정 파일명
status          VARCHAR(20) DEFAULT 'pending'
                -- 'pending' | 'applied' | 'error'
apply_error     TEXT                      -- 적용 실패 메시지
added_at        TIMESTAMP NOT NULL DEFAULT NOW()
applied_at      TIMESTAMP
sort_order      INT DEFAULT 0             -- 정렬 순서
```

```sql
id          SERIAL PRIMARY KEY
session_id  INT NOT NULL REFERENCES workspace_sessions(id) ON DELETE CASCADE

op_type     VARCHAR(30) NOT NULL
            -- 'tag_edit' | 'rename' | 'lrc_fetch' | 'cover_change'
```

## 5. 백엔드 API 설계

### 5-1. 신규 라우터: `workspace.py`

#### 세션 관리
```
GET  /api/workspace/session/current       현재 세션 (없으면 자동 생성)
POST /api/workspace/session/new           새 세션 시작 (기존 세션 초기화)
POST /api/workspace/session/{id}/apply    스테이징된 변경 사항 일괄 파일 반영
POST /api/workspace/session/{id}/discard  세션 폐기
```

#### 아이템 관리
```
POST /api/workspace/load-folder   폴더 불러오기 (현재 세션에 추가)
  Body: { folder_path, recursive: false }
  → 해당 폴더의 오디오 파일들을 세션에 추가

POST /api/workspace/load-files    파일 불러오기 (현재 세션에 추가)
  Body: { file_paths: [str] }

GET  /api/workspace/items         현재 세션 파일 목록
DELETE /api/workspace/items/{id}  개별 파일 제거

PUT  /api/workspace/items/{id}/stage-tags
  Body: { tags: { title, artist, ... } }
  → pending_tags 업데이트 (파일 미반영)

PUT  /api/workspace/items/{id}/stage-rename
  Body: { new_name: str }
  → pending_rename 업데이트

POST /api/workspace/items/{id}/apply
  → 단일 파일만 즉시 반영

GET  /api/workspace/items/{id}/diff
  → original_tags vs pending_tags 비교 반환
```

#### 히스토리
```
GET  /api/workspace/history                   세션 히스토리 목록 (페이지네이션)
GET  /api/workspace/history/{session_id}      세션 상세 (items + ops)
DELETE /api/workspace/history/{session_id}    히스토리 삭제
```

#### 라이브러리 피커 (기존 browse.py 엔드포인트 재사용)
```
GET /api/browse/roots        라이브러리 루트 폴더 목록 (기존)
GET /api/browse/children     하위 폴더 목록 (기존)
GET /api/browse/files        폴더 내 파일 목록 (기존, 읽기 전용 모드)
```

### 5-2. 적용 로직 상세 (`apply` 처리)

```python
def apply_session(session_id: int, db: Session):
    session = db.get(WorkspaceSession, session_id)
    items = session.items  # pending 상태인 것만

    results = []
    for item in items:
        errors = []

        # 1. 태그 변경 반영
        if item.pending_tags:
            try:
                tag_writer.write_tags(item.file_path, item.pending_tags)
                item.status = 'applied'
            except Exception as e:
                errors.append(str(e))

        # 2. 파일명 변경 반영
        if item.pending_rename:
            try:
                new_path = rename_file(item.file_path, item.pending_rename)
                item.file_path = new_path
            except Exception as e:
                errors.append(str(e))

        # 3. 작업 기록
        log_op(session_id, item.file_path, 'apply', {...})

        if errors:
            item.status = 'error'
            item.apply_error = '; '.join(errors)
        else:
            item.status = 'applied'
            item.applied_at = datetime.utcnow()

    session.status = 'applied'
    session.applied_at = datetime.utcnow()
    db.commit()

    return {"applied": applied_count, "errors": error_count}
```

### 6-5. LRC 검색 처리

LRC는 `.lrc` 파일 저장이 가역적이고 부작용이 없으므로:
- **즉시 저장** 방식 유지 (스테이징 불필요)
- 워크스페이스 아이템에 `has_lrc` 상태만 업데이트
- 히스토리에 `lrc_fetch` 작업으로 기록

## 7. 마이그레이션 전략

### 7-1. 기존 기능 유지

현재 Browser.vue의 직접 편집 기능은 **호환 모드**로 유지:
- 설정에 `workspace_mode: true/false` 옵션 추가
- 기본값: `true` (새 워크스페이스 모드)
- `false` 시: 기존 직접 편집 방식 유지 (기존 사용자 지원)

또는 완전 교체 방식:
- 기존 `/browse/write-tags` 등 API는 그대로 유지 (라이브러리 뷰, 앨범 상세 등에서 사용)
- Browser.vue만 워크스페이스 방식으로 전환

### 7-2. 기존 DB 데이터
