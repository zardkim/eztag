# eztag 백업 및 복원 계획

## 1. 500 오류 원인 분석

### 증상
설정 → 데이터 관리 → 백업 생성 클릭 시 HTTP 500 오류 발생.

### 원인
`pg_dump` 버전과 PostgreSQL 서버 버전 불일치.

- **서버**: `postgres:15` 이미지 (PostgreSQL 15.x)
- **Dockerfile 기존 설치**: `postgresql-client` (Debian 기본 패키지) → 버전이 서버와 다를 수 있음
- **결과**: `pg_dump`가 서버보다 높은 버전일 경우 연결 거부 또는 호환 오류 발생

### 로컬 개발 환경 참고
```
$ pg_dump --version
pg_dump (PostgreSQL) 16.13 (Ubuntu 16.13-0ubuntu0.24.04.1)
```
로컬에서 v16 `pg_dump`로 v15 서버에 접속하면 동일한 오류 재현 가능.

## 2. 수정 내용

### Dockerfile 변경 (`backend/Dockerfile`)

**이전:**
```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq-dev gcc \
    && rm -rf /var/lib/apt/lists/*
```
(기본 Debian 패키지로 `postgresql-client` 설치 → 버전 불확실)

**이후:**
```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq-dev gcc curl gnupg \
    && curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc \
       | gpg --dearmor -o /usr/share/keyrings/postgresql-archive-keyring.gpg \
    && echo "deb [signed-by=/usr/share/keyrings/postgresql-archive-keyring.gpg] \
       https://apt.postgresql.org/pub/repos/apt bookworm-pgdg main" \
       > /etc/apt/sources.list.d/pgdg.list \
    && apt-get update && apt-get install -y --no-install-recommends postgresql-client-15 \
    && rm -rf /var/lib/apt/lists/*
```

PostgreSQL 공식 APT 저장소에서 `postgresql-client-15`를 명시적으로 설치하여 서버 버전과 일치시킴.

### 적용 방법
Docker 이미지를 다시 빌드해야 수정 사항이 반영됩니다:
```bash
docker-compose down
docker-compose build backend
docker-compose up -d
```

## 3. 백업 대상

### 포함 항목

| 항목 | 이유 |
|------|------|
| PostgreSQL DB 전체 (`pg_dump`) | 모든 설정, 프리셋, 음악 메타데이터 포함 |

DB에 포함되는 테이블:

| 테이블 | 내용 |
|--------|------|
| `artists` | 아티스트 정보 |
| `albums` | 앨범 정보 |
| `tracks` | 트랙 메타데이터 (제목, 아티스트, 장르, 연도 등) |
| `scan_folder` | 스캔 대상 폴더 목록 |
| `app_config` | 모든 설정값 및 프리셋 (파일명 변경 패턴, Spotify 키, 스캔 간격 등) |
| `scan_log` | 스캔 이력 |

### 제외 항목

| 항목 | 이유 |
|------|------|
| 커버아트 (`data/covers/`) | 파일 크기가 크고, 스캔 재실행 시 자동 복구 가능 |

## 4. 백업 파일 구조

```
eztag_backup_YYYYMMDD_HHMMSS.tar.gz
├── db.dump          # pg_dump --format=custom 출력
└── backup_meta.txt  # 메타 정보 (버전, 타임스탬프, DB명)
```

### backup_meta.txt 예시
```
backup_version=3.0
timestamp=20260324_153000
db_name=eztag
```

### 저장 위치
Docker 볼륨: `./data/backup/` (컨테이너 내 `/app/data/backup/`)

---

## 5. 백업 생성 절차

1. UI: 설정 → 데이터 관리 → 백업 생성 클릭
2. 백엔드 `POST /api/backup/create` 호출
3. `pg_dump -h {host} -p {port} -U {user} -d {dbname} --format=custom -f db.dump`
4. `backup_meta.txt` 생성
5. 두 파일을 `tar.gz`로 압축
6. `data/backup/eztag_backup_YYYYMMDD_HHMMSS.tar.gz` 저장
7. 파일명을 응답으로 반환

---

## 6. 복원 절차

1. UI: 설정 → 데이터 관리 → 백업 파일 선택 → 복원 클릭
2. 백엔드 `POST /api/backup/restore/{filename}` 호출
3. `tar.gz` 압축 해제 → `db.dump` 추출
4. `pg_restore -h {host} -p {port} -U {user} -d {dbname} --clean --if-exists db.dump`
   - `--clean`: 복원 전 기존 객체 삭제
   - `--if-exists`: 객체가 없어도 오류 무시
5. 반환값: `{ ok: true, filename: "...", warnings: "..." }`

> **주의**: 복원은 현재 DB의 모든 데이터를 덮어씁니다. 복원 전 현재 상태를 백업하는 것을 권장합니다.

---

## 7. 커버아트 복구 방법 (복원 후)

커버아트는 백업에 포함되지 않으므로, 복원 후 스캔을 다시 실행하면 자동으로 재생성됩니다:

1. 설정 → 스캔 → 스캔 시작
2. 스캔 중 각 파일의 태그에서 커버아트를 추출하거나, 폴더의 `cover.jpg` / `folder.jpg` 파일을 자동 임베드

| 3.0 | DB만 포함 (커버아트 제외), pg_dump custom format |
